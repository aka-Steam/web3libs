### 4. Нагрузка на CPU и потребление памяти (браузер)

**Цель:** получить качественные и/или полуколичественные показатели нагрузки при длительных прогонах.

**Важно:** такие метрики обычно снимаются **вне** основного кода бенчмарка, с помощью инструментов профилирования браузера, чтобы не искажать измерения латентности.

**Процедура (типовая):**

1. Открыть инструменты разработчика, вкладка производительности (Performance).
2. Начать запись, запустить длительный бенчмарк (много повторов).
3. Остановить запись, выделить интервал, соответствующий нагрузочному сценарию.
4. Для памяти — вкладка Memory: снять снимок кучи до и после серии прогонов, сравнить объёмы и признаки утечек.

**Результат:** скриншоты профиля CPU и сравнение снимков памяти для каждой библиотеки при одинаковом сценарии.

### 5. Надёжность и обработка ошибок

**Цель:** зафиксировать поведение библиотек при типичных сбоях: недоступный RPC, смена сети, обрыв соединения.

**Процедура:**

1. **Неверный RPC:** намеренно указать недоступный адрес, выполнить базовый вызов (например, получение номера блока), записать текст и тип ошибки.
2. **Смена сети:** инициировать переключение сети через кошелёк (если применимо), зафиксировать успех или сообщение об ошибке.
3. **Обрыв узла:** остановить локальный узел во время прогона, зафиксировать реакцию приложения и характер сообщений.

**Результат:** качественное сравнение формулировок ошибок и удобства диагностики; при необходимости — таблица «сценарий — наблюдаемое поведение — цитата ошибки».
