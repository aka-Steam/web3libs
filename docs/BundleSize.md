
Кратко по шагам и по отличию от «размера npm-пакета».

## Как получилось сравнение «именно библиотек»

1. **Отдельные точки входа** (`index-lib-ethers.html` / `index-lib-viem.html` / `index-lib-web3.html`) ведут только на `src/lib-bundles/ethers-entry.ts`, `viem-entry.ts` или `web3-entry.ts`. В этих сборках **нет React, нет вашего UI и бенчмарка** — только граф зависимостей, который вытягивают импорты из `ethers`, `viem` или `web3`.

2. **Те же *value*-импорты, что в адаптерах** — то есть не «весь репозиторий целиком», а **подмножество API**, которое вы реально подключаете в `ethersAdapter` / `viemAdapter` / `web3Adapter` (провайдер, клиенты, `http`/`custom`, хеши, сериализация и т.д.). Bundler (Rollup через Vite) тянет **только достижимый по импортам код** плюс его внутренние зависимости в графе.

3. **Побочный эффект на `globalThis`** — без него Rollup мог бы считать «entry ничего не делает» и выкинуть граф. Запись в `globalThis.__web3LibBundleProbe` фиксирует ссылки на импорты так, чтобы **фактически попал вельный код библиотеки**.

4. **Сборка как в продакшене** — минификация, объединение чанков по правилам Vite: вы меряете **размер того JS, который реально получился на выходе** (все `.js` из `assets/`, суммарно).

Итоговая метрика «library bundles» — это **«сколько весит заbundленый, заминифицированный граф для вашего набора импортов, без приложения»**, а не абстрактная цифра «пакет на npm».

## Совпадает ли это с «размером npm-пакета»?

**Нет, не равно.** Разные вещи:

| Что вы меряете (library probe) | Что часто имеют в виду под «размером пакета» |
|--------------------------------|---------------------------------------------|
| Итог **прод-сборки** (one/tree-shaken graph) под **конкретные импорты** | Размер **архива на npm** / папки `node_modules/ethers` целиком |
| Уже после **tree-shaking** под ваш entry | **Декларативный** размер (все экспорты, все файлы в пакете) |
| **Минифицированный** JS | Исходники + source maps + типы + README в tarball |
| Может быть **несколько чанков** (как у viem с `ccip`), все суммируются | Bundlephobia и т.п. часто показывают **другой** entry/другой сценарий |

То есть вы честно сравниваете **«ethers / viem / web3 при таком использовании в нашем проекте»** при **одинаковом уровне покрытия в адаптерах**. Это ближе к **реальной стоимости в прод-сборке**, чем к строке «сколько весит tgz с npm».

Если бы вы импортировали из `ethers`/`viem` другой набор символов, цифры изменились бы — потому что это не «вес всего пакета», а **вес достижимого подграфа**.

## Зачем тогда отдельно «full application»

Там в сумму входят React, ваш код и один стек — это метрика **«как приложение уедет пользователю»**; доля библиотеки в ней размыта общим «фундаментом».

**Краткий вывод:** сравнение library bundles — это **хорошо согласованное, tree-shake’нутое, сопоставимое между собой** измерение двух стеков **в одном и том же сценарии импортов**; оно **не тождественно** «размеру пакета на npm в целом».


>Под чанками (chunks) имеются в виду отдельные файлы .js, которые Vite/Rollup создаёт при сборке. Браузер может грузить не один огромный бандл, а несколько кусков: например, один основной (index-*.js) и маленькие дополнительные.
>
>Зачем так делают
>
>Вынести код, который нужен не всем или подгружается позже (code splitting).
>Дать кэшированию шанс: поменялся только один чанк — перекачивается не всё приложение.
>Что за ccip у viem
>
>В графе зависимостей viem есть модуль, связанный с CCIP (Cross-Chain Interoperability Protocol / cross-chain в контексте viem). При ваших импортах (createPublicClient, wallet, http и т.д.) часть общей инфраструктуры viem попадает в отдельный чанк, и Rollup называет такие файлы по хэшу/фрагменту пути — у вас в выводе он выглядит как ccip-….js (~2–3 KB).
>
>То есть это не «ещё одна библиотека с npm», а фрагмент вашего же viem-бандла, просто вынесенный в отдельный файл. Ваш скрипт суммирует размеры всех .js в assets/, поэтому и основной чанк, и ccip учитываются вместе.
